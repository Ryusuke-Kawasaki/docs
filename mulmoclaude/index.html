むずかしい論文4本 → レストランの例ひとつで

AIに、自分専用の台帳を
作らせるという話。

「ワークスペースがエージェントになる」「アプリがデータになる」—— 抽象的でピンとこない4本の論考を、1軒のレストランを記録するという 具体例だけで、最初から最後まで通して説明します。

専門用語ほぼゼロ フォルダが育つのを追う 出典：receptron/mulmoclaude
先に結論だけ

ソフトは「みんなに売る製品」から、あなた専用に、その都度AIが育てるものへ変わる。 コードは使い捨てでいい。大事なのは「何をしたいか（意図）」と「あなたのデータ」だけ。

——この一文の意味を、下のレストランの例で順番にほどいていきます。

なぜ

「いま」これが可能になったのか

昔は、ソフトを作るのは高くつきました。だから1個作って何百万人に使ってもらう＝ 「平均的なみんな」に合わせるしかなかった。「自分のレストランを東京の路線図みたいに表示するアプリ」なんて、 欲しい人が世界に1人しかいないので、どの会社も絶対に作りません。

昔

みんなのために作る

作るのが高い → 大勢に薄く配って元を取る → 一人ひとりの「ちょうど欲しい形」は永遠に作られない。

いま

あなた一人のために作る

AIが頼んだ瞬間にコードを書く → 一人のために作るのが、ほぼタダ。あなたはコードを一度も見ない。

「一人のために作る」が「百万人のために作る」のと同じくらい安くなった瞬間、 わざわざ平均に合わせる理由が消えます。これが4本の論文に共通する出発点です。

本編

フォルダが育つのを、順番に追う

最初、あなたのフォルダは空っぽです。ここから、たった一言で何のファイルが生まれるかを見ていきます。

1
行きたいレストランを管理して

AIは「台帳の形」を1枚書く＝ アプリ完成

AIが大層なプログラムを書くわけではありません。「店名・エリア・期待度の欄がある台帳」と 欄を定義する小さなファイルを1枚書くだけ。これだけでアプリは完成です。 新品の手帳は、まだ何も書いてなくても「手帳」ですよね。それと同じ。

📋restaurants/schema.json＝ 台帳の欄の定義▶ 
{
  "title": "行きたいレストラン",
  "fields": {
    "name":   { "type":"string", "label":"店名" },
    "area":   { "type":"string", "label":"エリア" },
    "rating": { "type":"number", "label":"期待度" }
  }
}
この「欄を決めるだけでアプリ」が collections-architecture.md
2

AIは「操作の手順書」も1枚書く＝ skill

AIが次からこの台帳をどう扱うかを書いた、ただのメモです。プログラムではなく指示文。 本物のレシピ用スキルも、中身は「友達みたいに振る舞って。ファイルパスの話はユーザーにしないで」といった 普通の文章でした。

📝restaurants/SKILL.md＝ 操作手順のメモ▶ 
---
name: restaurants
description: 「○○に行きたい」「リスト見せて」のとき使う
---
# 行きたいレストラン
- 保存：<id>.json を書く
- 一覧：presentCollection を restaurants で呼ぶ
いまのフォルダ
📁 restaurants/
  ├ schema.json  # 台帳の形（アプリ本体）
  └ SKILL.md    # 操作手順NEW
3
恵比寿のフレンチ、リオンに行きたい

台帳に1件だけ書き足す＝ 使う

ここが大事な点でした。データは最初から要りません。 空の台帳から始まって、あなたが使うたびにAIがファイルを1個ずつ足していく。だんだん溜まっていきます。

🍽️restaurants/lyon.json＝ データ1件目▶ 
{ "id":"lyon", "name":"リオン", "area":"恵比寿", "rating":5 }
いまのフォルダ
📁 restaurants/
  ├ schema.json
  ├ SKILL.md
  └ lyon.jsonNEW
4
行ったかどうかも記録できるようにして

schema を1行書き換えるだけ＝ 育てる

普通のソフトなら、機能追加はエンジニアがコードを書いて → ビルドして → 再デプロイ、の大仕事。 でもここではAIが台帳の定義に1行足すだけ。デプロイも再起動もありません。 台帳がその場で進化します。

✏️schema.json に1行追加＝ デプロイの壁なし▶ 
"fields": { … 既存の欄 …,
  "visited": { "type":"boolean", "label":"行った？" }  ← これを足すだけ
}
「機能追加＝文章を1枚書き足す＝AIが自分をバージョンアップ」が workspace-is-the-agent.md
5
一覧じゃなくて地図で見たい

AIが見た目（HTML）を1枚生やす＝ あなたは中身を見ない

表形式じゃ物足りないとき、AIがHTMLを書いて台帳に足します。あなたはそのコードを一度も見ません。 気に入らなければ「色を変えて」と言えば捨てて作り直す。修理はしない。

🗺️restaurants/views/map.html＝ 自分専用UI（中身は見ない）▶ 
<!-- AIが書いた地図表示。あなたは開かない。
     不満なら「色変えて」で丸ごと作り直す（=使い捨て） -->
<div id="map">…</div>
「コードは使い捨て・あなたは見ない」が software-for-one.md
整理

つまずきやすい2つの言葉

「台帳1個の話」と「台帳を全部しまった引き出しの話」を、混ぜないのがコツです。

📋

Collections ＝ 台帳1個

collections-architecture.md

レストラン台帳をどう作るか。欄を決めるだけでアプリになり、専用のプログラムは要らない、という1個の仕組みの話。

🗄️

Workspace ＝ 引き出し全体

workspace-is-the-agent.md

たくさんの台帳に加えて、AIの性格設定やできること（スキル）まで全部入ったフォルダ＝AI本体の話。

その「引き出し」には、台帳だけでなく次の2種類のファイルも入っています。どちらもただの文章です。

roles
性格設定「丁寧な秘書として、結論から言う」のような、AIのキャラや口調を決めた文章。書き換えれば性格が変わる。
skills
できること（手順書）「請求書を作るときの段取り」のような作業手順のメモ。AIが必要なとき読んで、その通り動く。1枚足す＝新しい仕事ができる。
📁 ワークスペース # ＝ あなたのAI本体まるごと ├ roles/ # 性格設定（文章） ├ skills/ # できることの手順書たち ├ restaurants/ # レストラン台帳 ├ shopping/ # 買い物リスト └ cooking/ # レシピ帳

全部ただのファイルなので、cp -r でコピーすればAIごと引っ越せるし、git で履歴も残る。 このフォルダこそがAIそのもの、というのが一番言いたかったことです。

まとめ

結局やっているのは、3種類のファイル操作だけ

難しい4本の論文も、具体には次の3つに集約されます。

schema を書く
アプリを作る欄を決めた台帳の定義を1枚書く。＝ collections
json を足す / 直す
使う・育てるレコードを1件足す、台帳の定義を書き換える。＝ workspace（自己改善）
view.html を書く
自分専用の見た目を生やすその都度UIを作り、不満なら捨てて作り直す。＝ software-for-one
ひとことで。 あなたは「何が欲しいか」を言うだけ。AIが台帳とその見た目をその場で作り、使ううちに育ち、フォルダごとAIに溜まっていく。 ——コードを書く人も、見る人も、もういません。

元になった4本の論考
